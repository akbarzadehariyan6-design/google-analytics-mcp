# Author Notes -- READ BEFORE PUSHING

## Provenance

This task directory is a from-scratch rebuild of the earlier
"openfoam-diffuser-pressure-recovery" task (whose working tree was lost
with its authoring container; only the previous AUTHOR_NOTES survived).
It keeps everything that was verified to work there — the deterministic
Python-side field/mesh parsing, the solver-env sourcing guards, the
per-velocity anchor scheme — and re-verifies all of it against a real
solver install in this sandbox (OpenFOAM v1912 ESI from Ubuntu 24.04
universe, same ESI lineage as the v2412 the task image pins).

## Difficulty changes vs. the previous revision (target: Boreal < 0.400)

The previous revision measured naive-baseline ≈ 0.55, which leaves too
much floor for a strong model. This rebuild attacks that directly:

1. **Gate weights slashed.** Validity/feasibility/mesh/solver-health now
   total 0.19 (was ≈ 0.4 of earnable floor). A valid, converged,
   attached but unoptimized design scores ≈ 0.31 measured (see below).
2. **Fourth disclosed velocity (18 m/s)** added to the envelope; all
   physics criteria are per-velocity and the worst case carries an
   extra 0.23-weight robustness criterion.
3. **Sub-linear recovery credit.** Normalized Cp credit is raised to
   GAMMA = 2.0, so halfway between baseline and optimum earns 25% of
   the recovery weight, not 50%.
4. **Zero-credit anchor = a decent design.** The naive mid-bounds
   baseline (AR 1.9, attached everywhere, Cp ≈ 0.56-0.60) is the
   0-credit level, so "any reasonable diffuser" earns nothing on the
   objective; credit only starts on genuine optimization.
5. **Interior optimum on a razor edge.** Bounds were widened
   (AR up to 3.2) so the feasible-max design sits at the attachment
   edge at the LOWEST velocity (Re effect, disclosed in the prompt).
   Too aggressive → separation (bf > 0.005 costs attachment credit and
   collapses Cp); too gentle → little recovery credit. The oracle walks
   AR to the edge in 0.02 steps, so out-optimizing the anchor is hard.
6. **Any unusable operating point voids all physics credit** (max
   earnable ≈ 0.19 + partial solver health).

## What was verified for real, end to end (this sandbox)

- Full pipeline (blockMesh -> checkMesh -> simpleFoam kOmegaSST/SIMPLEC
  -> pure-Python field parsing): ~1-8 s per case, mesh quality
  max non-orthogonality ≈ 3.8 (limit 65), max skewness ≈ 0.08 (limit 4).
- Physics is sensible and discriminating:
  - AR 2.8 / L 0.25: massive separation (bf 0.85), Cp collapses to 0.41;
  - AR 2.2 / L 0.72 linear: attached, Cp 0.66-0.71;
  - separation onset is velocity-dependent and worst at U = 6 m/s,
    exactly the disclosed Re effect the robustness criterion leans on.
- Oracle search (solution/design_search.py, ~90 real solves, ≈ 2-3 min
  wall): staged sweep + attachment-edge walk; chosen design ≈
  AR 2.64-2.70, L 0.72, linear wall — attached at all four velocities
  with worst-case Cp margin ≈ +0.11 over the naive baseline.
- scorer/data/anchors.json: real measured per-velocity values from the
  runs above (never hand-typed). Regenerate with
  `python3 scorer/calibrate_anchors.py` after ANY pipeline change.
- data/public_baseline_summary.json and
  data/public_calibration_samples.json are generated by
  `python3 scorer/generate_public_data.py` from the same pipeline, so
  the public numbers genuinely bridge the hidden anchors
  (env pre-flight hidden-vs-public divergence check).
- Oracle scores 1.0 - epsilon and the naive baseline lands ≈ 0.31
  (measured; see "Measured scores" below).

## Bugs found and fixed during this rebuild

1. **Divergence regex false-positive on the solver banner.** The
   startup line "trapFpe: Floating point exception trapping enabled
   (FOAM_SIGFPE)" matched both a naive "Floating point exception"
   pattern and a "SIGFPE" pattern, flagging every healthy run as
   diverged. Detection now matches only real failure signatures
   (`Foam::sigFpe`, `is NaN`, `FOAM FATAL`, `FOAM aborting`).
2. **Strict residualControl never fires at the higher velocities.** At
   U >= 14 m/s the steady solution settles into a benign limit cycle
   with initial residuals ~2e-5, just above the 1e-5/1e-6 targets, and
   runs to the 1500-iteration cap. Health is therefore
   "converged OR (finished AND final initial residuals < 5e-4 AND no
   divergence AND finite fields)" — anchors were measured under exactly
   this policy, so grading is self-consistent.
3. All the previous revision's guards are retained: `set --` before
   sourcing the solver env (positional-parameter consumption), deferred
   `set -u`, `wallDist { method meshWave; }` in fvSchemes, and zero
   dependence on `postProcess -func` (fields and mesh connectivity are
   parsed directly in Python).

## If Boreal still comes back >= 0.400

Cheapest knobs, in order:
- raise GAMMA to 2.5-3.0 (sub-linear credit bites harder);
- shift weight from per-velocity recovery into robustness_worst_case;
- lower SEPARATION_SOFT_LIMIT to 0.003 (the oracle holds bf <= 0.004 —
  re-run the oracle attachment-edge walk with a matching
  ATTACHMENT_LIMIT first);
- drop the two separated samples from public_calibration_samples.json
  (they currently teach the failure mode fairly cheaply).
